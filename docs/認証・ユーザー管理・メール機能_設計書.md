# 認証・ユーザー管理・メール機能 設計書

## 概要
ユーザー認証、プロフィール管理、注文時のメール通知機能を実装する。

## 機能要件

### 1. ユーザー認証
- ✅ メールアドレスとパスワードでログイン
- ✅ ログアウト機能
- ✅ セッション管理（JWT）
- ✅ パスワードのハッシュ化（bcrypt）

### 2. ユーザープロフィール管理
- ✅ プロフィール登録（名前、住所、電話番号）
- ✅ プロフィール編集
- ✅ プロフィール表示

### 3. 注文との統合
- ✅ 注文時にログインユーザーの情報を使用
- ✅ 注文履歴でユーザーの住所を表示
- ✅ ユーザー別の注文フィルタリング

### 4. メール通知
- ✅ 注文確認メール（OrderPlacedイベント経由）
- ✅ 請求書メール（OrderPlacedイベント経由）

**アーキテクチャ**: イベント駆動
- 注文作成時に`OrderPlaced`ドメインイベントを発行
- イベントハンドラーがイベントをリスンしてメール送信を実行
- 関心の分離により、注文ロジックとメール送信が疎結合

**ファイル**: `openapi/openapi.yaml`

**認証API**:
```yaml
# POST /api/auth/register - ユーザー登録
#   Request: { email: string, password: string }
#   Response: { user: UserDto, token: string }
# POST /api/auth/login - ログイン
#   Request: { email: string, password: string }
#   Response: { user: UserDto, token: string }
```

**ユーザーAPI**:
```yaml
# GET /api/users/me - 自分のプロフィール取得（要認証）
#   Headers: Authorization: Bearer {token}
#   Response: UserDto
# PUT /api/users/me - プロフィール更新（要認証）
#   Headers: Authorization: Bearer {token}
#   Request: UpdateProfileRequest (name, address, phone)
#   Response: UserDto
```

**注文API（修正）**:
```yaml
# POST /api/orders - 注文作成（要認証に変更）
#   Headers: Authorization: Bearer {token}
#   Request: CreateOrderRequest（userIdフィールドは不要、トークンから取得）
#   Response: OrderDto（userIdフィールド追加）
# GET /api/orders - 注文一覧取得（要認証に変更、自分の注文のみ取得）
#   Headers: Authorization: Bearer {token}
#   Response: OrderDto[]（自分のuserIdでフィルタリング済み）
```

**スキーマ定義**:
```yaml
# UserDto (id, email, profile, createdAt, updatedAt)
# LoginRequest (email, password)
# RegisterRequest (email, password)
# AuthResponse (user, token)
# UpdateProfileRequest (name, address: AddressDto, phone)
# AddressDto (postalCode, prefecture, city, addressLine)
# OrderDto (userIdフィールド追加)
```

**セキュリティスキーム**:
```yaml
# bearerAuth (type: http, scheme: bearer, bearerFormat: JWT)
```

- **疎結合**: CreateOrderユースケースはメール送信について知らない
- **拡張性**: 将来的に他のイベントハンドラー（在庫更新、分析など）を追加可能
- **信頼性**: メール送信失敗しても注文処理は完了する

## 技術スタック

### バックエンド
- **認証**: JWT (jsonwebtoken)
- **パスワードハッシュ化**: bcrypt
- **データベース**: BunビルトインSQLite (bun:sqlite)

### フロントエンド
- **状態管理**: React Context API
- **ストレージ**: localStorage（トークン保存）

## この設計書の使い方

この設計書を以下のように活用してください：

1. **タスク単位で実装**:
   - 各Phaseの各Taskを順番に実装
   - 各Taskは独立しており、TDD方式で進行

2. **Claude への指示例**:
   ```
   「Phase 0 を実装してください」（最初に実行）
   「Phase 1の Task 1-1 を実装してください」
   「Phase 2の Task 2-1 のテストを書いてください」
   「Phase 4まで完了したので、Phase 5に進んでください」
   ```

3. **契約駆動開発の順守**:
   - **Phase 0を最初に実行**: すべての実装に先立ちOpenAPI仕様書を作成
   - 型生成後、すべてのフェーズで生成された型を使用
   - Swagger UIで仕様を確認しながら開発

4. **進捗管理**:
   - 各Phaseの✅マークで進捗を管理
   - 完了したタスクにチェックを入れる

5. **段階的なコミット**:
   - 各Phase完了時にコミット
   - コミットメッセージに「Phase X完了」を明記

## 完成イメージ

### ユーザーフロー
1. ユーザーが `/register` で登録 → SQLiteにユーザー情報保存
2. 自動的にログイン状態に
3. `/profile` でプロフィール登録 → SQLiteに更新
4. カートから注文 → プロフィールの住所が自動入力
5. 注文完了 → `OrderPlaced`イベント発行 → イベントハンドラーがメール送信（MailHogに送信）
6. メール確認 → MailHog Web UI（http://localhost:8025）で注文確認メールと請求書を確認
7. `/orders` で注文履歴確認 → 自分の住所が表示

**イベントフロー図**:
```
CreateOrderユースケース
  ↓
Order作成成功
  ↓
OrderPlacedイベント発行
  ↓
EventDispatcher
  ↓
OrderPlacedHandler（非同期）
  ├→ UserRepository.findById()
  ├→ EmailService.sendOrderConfirmation()
  └→ EmailService.sendInvoice()
       ↓
    MailHog SMTP
```
